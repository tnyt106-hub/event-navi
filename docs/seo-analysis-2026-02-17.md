# SEO分析レポート（2026-02-17）

## 対象
- 静的公開物: `docs/**/*.html`
- サイトマップ: `docs/sitemap*.xml`
- 既存の自動チェック: `scripts/common/check-seo-required.js`

## 結論（要約）
- 現状、**SEOの必須メタ欠落はほぼ解消済み**で、技術的な土台は安定しています。
- 一方で、検索流入の伸びしろとしては、次の3点が優先です。
  1. **`/spot/index.html` の noindex + canonical 依存を縮小し、URL設計を一本化**する
  2. **メタディスクリプションが短いページの改善（特に施設詳細）**
  3. **`noindex` が付いた日付ページ群の運用KPIを定義し、意図どおりに効いているかを計測する**

---

## 確認結果

### 1) 必須SEO要素チェック
- `check-seo-required.js` の実行結果は **507ファイルでOK**。
- `title` / `description` / `canonical` / `og:*` / `twitter:*` / JSON-LD必須ページの検査は通過。

### 2) HTML実測チェック（機械集計）
- 検査対象: 505 HTML（所有権確認ファイル・部分テンプレートを除外）
- `h1` 欠落: 0
- `h1` 複数: 0
- `description` 欠落: 0
- `title` 重複: 0
- `canonical` 重複URL: 1件（`https://event-guide.jp/facility-name/`）
- `noindex` ページ: 387（`date` 配下386 + `spot/index.html` 1）

### 3) 確認できた課題（優先順）

#### 優先度A: `/spot/index.html` の正規化設計
- `docs/spot/index.html` は `noindex,follow` かつ canonical が `/facility-name/` を向いています。
- これは意図としては理解できますが、
  - 内部リンクや共有URLで `/spot/` 単体が参照されると、検索評価の解釈が分散しやすい
  - canonical重複（同一正規URLへの集中）が運用上の混乱ポイントになりやすい
- **改善案**
  - `/spot/` を恒久的に使わないなら、`/facility-name/` へ 301 リダイレクト化（最も明確）
  - 使うなら canonical を自己参照に戻し、ページ役割を再定義

#### 優先度A: ディスクリプション文字数の最適化
- 施設詳細で 40〜50文字台の `description` が複数あり、検索結果スニペットで情報量が不足しやすい状態です。
- **改善案**
  - 目安 60〜90文字（日本語）で「地域 + 施設種別 + 何が分かるか」を明記
  - 生成スクリプトで最小文字数ルール（例: 60文字未満は警告）を追加

#### 優先度B: `noindex` 日付ページ群の運用妥当性
- `date` 配下に `noindex` が多数付与されています（386ページ）。
- サイトマップには noindex URL が混在しておらず、実装整合性は取れています。
- ただし、SEO成果としては「意図どおりの絞り込み」か「機会損失」かの判定が必要です。
- **改善案**
  - Search Consoleで、`/date/` の表示回数・CTR・平均順位を「index対象日」と比較
  - noindex基準を「イベント件数」「日付の近接性」で定量化

#### 優先度B: タイトルが汎用的なハブページの意図明確化
- `docs/spot/index.html` のタイトルは「施設詳細」で、検索語との一致が弱い可能性があります。
- **改善案**
  - 301運用しない場合は、ページ名を「施設詳細（URL直アクセス用）」など用途明示へ調整

## すぐ着手できる改善タスク（低コスト順）
1. `/spot/` の扱いを「301統合」か「独立ページ運用」か意思決定
2. 施設詳細の短い `description` から順に60文字以上へ拡張
3. `check-seo-required.js` に「description最小長」「title最小長」の警告モードを追加
4. Search Consoleで `/date/` noindex方針のKPI監視を開始
