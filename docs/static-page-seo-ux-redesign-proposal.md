# 静的ページ刷新案（SEO・ユーザビリティ重視 / 段階実行版）

## 1. 目的
- 静的ページ（トップ・施設一覧・スポット詳細・日付別ページ）の情報設計を見直し、**検索流入を増やす**。
- 初回訪問ユーザーでも迷わず目的の情報（開催場所・日程・アクセス）に到達できるようにし、**離脱率を下げる**。
- 一気に改修せず、**優先度の高いものから小さく実装→計測→次ステップ確定**で進める。

## 2. 進め方の基本方針（今回の追加要件）
- 変更は「Step 1〜Step 5」に分割し、各Stepは**1テーマだけ**を扱う。
- 各Stepは以下の順序で進める。
  1. 詳細設計を確定（対象ページ、文言、HTML構造、影響範囲）
  2. 最小改修を実装
  3. 計測（SEO/UX）
  4. 結果レビュー後に次Stepへ進む
- 各Stepごとに**完了条件（Definition of Done）**を設定し、完了条件を満たさない場合は次に進まない。

## 3. 全体優先順位（高→低）
1. メタ情報・見出し・パンくずの統一（検索理解の土台）
2. 主要導線（CTA）と0件時体験の改善（離脱防止）
3. 構造化データ（BreadcrumbList / Event / Place）
4. 内部リンク・回遊導線の強化
5. OGP/Sitemap/CWVなどの運用最適化

---

## 4. ステップ別計画（詳細化しながら段階実行）

### Step 1（最優先）: メタ情報・見出し・パンくずの統一
**狙い**
- 検索エンジンにページの役割を正しく伝えるための基盤を最短で整える。

**対象ページ**
- `docs/index.html`（トップ）
- `docs/date/index.html`（日付）
- `docs/facility/index.html`（施設）
- `docs/spot/index.html`（スポット）

**作業項目**
- `<title>`, `<meta name="description">`, `<h1>` の命名規則を固定。
- パンくず（トップ > 一覧 > 詳細）をページ共通で導入。
- ページ内でHタグの階層飛び（H1→H3など）を解消。

**Step 1の完了条件（DoD）**
- 上記4ページでタイトル・説明・H1の命名規則が統一されている。
- パンくずが全対象ページで表示され、リンク切れがない。
- 主要ページの検索スニペット文言が重複しない。

**計測項目**
- Search Console: 対象ページの平均掲載順位 / CTR（2〜4週間）

---

### Step 2: 主要CTAと0件時導線の改善
**狙い**
- ファーストビュー離脱と「情報が見つからない離脱」を減らす。

**対象ページ**
- `docs/index.html`（3導線カード）
- `docs/date/index.html`（0件時提案）

**作業項目**
- トップに「日付で探す / エリアで探す / 施設で探す」の固定導線を配置。
- 日付ページ0件時に「別日候補」「近隣施設」「トップへ戻る」を表示。
- モバイルで主要CTAを押しやすいサイズ・位置へ調整。

**Step 2の完了条件（DoD）**
- 主要CTAの視認性がファーストビューで担保される。
- 0件時に次アクションが最低2つ以上提示される。
- モバイルでCTAの誤タップ率を下げるレイアウトになっている。

**計測項目**
- CTAクリック率
- 0件ページからの再遷移率
- 直帰率の変化

---

### Step 3: 構造化データ（JSON-LD）導入
**狙い**
- ページ内容を検索エンジンが機械的に理解しやすい状態へ引き上げる。

**対象ページ**
- 日付ページ: `Event`
- 施設/スポット詳細: `Place`
- 全詳細系ページ: `BreadcrumbList`

**作業項目**
- JSON-LDのテンプレートをページ種別ごとに作成。
- 必須項目（name, url, startDate, location等）を欠損なく出力。
- Rich Results Testでエラー0を目指す。

**Step 3の完了条件（DoD）**
- 対象ページで構造化データの必須項目が揃っている。
- テスト結果で重大エラーがない。

**計測項目**
- Search Consoleのリッチリザルト関連指標
- インデックスカバレッジの警告件数

---

### Step 4: 内部リンクと回遊導線の強化
**狙い**
- クローラビリティとユーザー回遊を同時に改善する。

**対象ページ**
- `docs/date/index.html`
- `docs/facility/index.html`
- `docs/spot/index.html`

**作業項目**
- 関連イベント、近隣スポット、前後日リンクの追加。
- 詳細ページ下部に「次に見る候補」をカード表示。
- 孤立ページ（内部リンク流入0）を減らす。

**Step 4の完了条件（DoD）**
- 主要ページに最低2系統以上の内部リンク導線がある。
- 主要導線のリンク切れがない。

**計測項目**
- 1セッションあたり閲覧ページ数
- 内部リンククリック率

---

### Step 5: 運用最適化（OGP/Sitemap/CWV）
**狙い**
- 更新運用を安定化し、継続的に検索評価を落とさない状態を作る。

**対象**
- `docs/sitemap.xml`
- 各ページhead内OGP/Twitter Card
- CSS/画像の軽量化ポイント

**作業項目**
- OGP/Twitter Cardをページ単位で最適化。
- `sitemap.xml`の更新漏れ検知フローを作成。
- LCP画像のサイズ最適化、不要CSS整理。

**Step 5の完了条件（DoD）**
- 新規ページ追加時のメタ・サイトマップ更新手順が文書化されている。
- CWV悪化要因の上位項目に対応済み。

**計測項目**
- Core Web Vitals（LCP/INP/CLS）
- SNS経由のクリック率

---

## 5. 各ステップ開始前に必ず決めること（詳細化テンプレート）
各Step着手前に、以下を5〜10分で合意してから実装する。

1. 対象ページ（どこを触るか）
2. 変更項目（何をどう変えるか）
3. 非対象項目（今回は何をやらないか）
4. 成功指標（何が上がれば成功か）
5. ロールバック条件（悪化時に戻す条件）

> このテンプレートを使うことで、都度「詳細を詰めてから改修」が可能になる。

## 6. 直近の実行提案（次に着手する順）
- まずは **Step 1** のみ実施（メタ情報・見出し・パンくず統一）。
- Step 1の結果を確認後、**Step 2** の詳細（CTA文言・0件時文言）を詰める。
- その後、Step 3以降を同じ進め方で繰り返す。

## 7. ヘッダーと背景の色差が気になる場合のデザイン改良案

現状は、ページ全体が寒色グラデーション背景（`--bg-gradient`）で、ヘッダーが単色アクセント（`--accent`）のため、
**「ヘッダーが背景から浮きすぎる」または「切り替わりが唐突に見える」** 体験が起きやすい。

ここでは、既存構造を活かしつつ段階的に試せる3案を示す。

### 案A（最小変更・推奨）: ヘッダーを半透明サーフェス化して背景と馴染ませる
**狙い**
- いまの配色資産を崩さず、境界のきつさだけを緩和する。

**変更イメージ**
- `--header-bg` を白系半透明（例: `rgba(248, 250, 252, 0.88)`）へ。
- ヘッダー文字色は白ではなく濃色（`--text`）へ寄せ、可読性を維持。
- `backdrop-filter: blur(6px)` を補助的に使う（非対応ブラウザでも致命的にならない設計）。

**メリット**
- 実装コストが低い。
- 視覚ノイズを減らし、上部UIが柔らかく見える。

**注意点**
- 透明度を上げすぎると文字が埋もれるため、コントラスト比（最低4.5:1）確認が必要。

### 案B（中変更）: 背景グラデーションとヘッダーを同一トーンで再設計
**狙い**
- ページ全体の色温度を統一し、デザインに一体感を持たせる。

**変更イメージ**
- 背景グラデーションを「彩度を少し抑えた青系」に再定義。
- ヘッダーは同グラデーションの最濃色を使う（同系色の明度差のみで構成）。
- CTAボタンは補色ではなく、同系の明るい青で差を作る。

**メリット**
- ブランド感が出る。
- 見た目の整合性が高く、長期運用向き。

**注意点**
- 一括調整になるため、地図カード・フッター・ボタンまで再検証が必要。

### 案C（大変更）: ヘッダーを白ベースに戻し、背景をコンテンツ領域だけに限定
**狙い**
- 情報サイトとしての可読性と「公式感」を最優先にする。

**変更イメージ**
- ヘッダーを `--surface` ベース（白）+ 濃色テキストへ。
- グラデーションはメイン背景またはヒーロー領域だけに適用。
- パンくず・見出し・CTAを青で統一し、情報優先UIに寄せる。

**メリット**
- 文字可読性が安定し、SEO文脈の「情報閲覧体験」と相性が良い。
- 将来コンテンツが増えても破綻しにくい。

**注意点**
- 現在のビジュアル印象（華やかさ）は少し下がる可能性がある。

### どれから試すべきか（実行順）
1. まず案AでA/B比較（最短で違和感を減らせる）。
2. 数値改善（直帰率・CTAクリック率）が小さければ案Bへ。
3. 「情報サイトとしての信頼感」を最優先する段階で案Cを検討。

### 計測指標（色調整の評価軸）
- ファーストビュー離脱率
- ヘッダー内CTAクリック率
- モバイルでのスクロール開始率
- Lighthouse Accessibility（コントラスト）

### 参考: まず試す配色トークン（案Aの初期値）
- `--header-bg: rgba(248, 250, 252, 0.88)`
- `--header-text: #1F2937`
- `--header-subtext: #475569`
- `--header-border: rgba(148, 163, 184, 0.35)`

> 補足: この初期値は「地図が主役」の現行構成を保ちながら、ヘッダーだけ違和感を下げるための出発点。
